## Git 工作流

Commit 类型约定：

```text
feat: 新功能
fix: Bug 修复
docs: 文档变更
refactor: 代码重构
chore: 构建、配置或维护项变更
```

### 提交信息规范

- Git 提交标题统一使用中文，避免只写英文或中英混杂的简略描述。
- 提交标题保持 `type: 简要说明` 结构，例如：`fix: 修复 ai_kitchen 墙体拖拽后房间未及时重算的问题`。
- 标题要明确说明“改了什么”或“解决了什么问题”，不要使用 `update`、`modify`、`fix bug` 这类信息量过低的表述。
- `type` 继续沿用仓库已有约定：`feat`、`fix`、`docs`、`refactor`、`chore`。
- 只要改动不是一次极小的单文件微调，就应该补充详细的提交正文，不要只有一行标题。
- 提交正文建议尽量使用稳定小标题，至少写清楚以下内容：
  - 问题描述 / 功能描述：当前问题是什么，或者这次新增的能力是什么。
  - 修复思路 / 实现思路：为什么选择当前方案，核心判断依据是什么。
  - 解决方案 / 实现方案：具体改了哪些模块、链路、文档或规则。
  - 影响范围：这次改动主要影响哪些目录、功能、交互或使用方式。
- 如果改动跨多个目录、涉及跨包调用链、影响行为较多，正文应写得更详细，帮助后续同学通过 git 历史快速理解上下文。
- 文档类提交也遵循同样原则，不要只写“补充文档”，应写明补的是哪类文档、为了解决什么问题、入口如何调整。
